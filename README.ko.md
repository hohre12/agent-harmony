# Agent Harmony

### 코드를 짠 AI가 직접 리뷰하면, 그건 리뷰가 아닙니다.

Claude Code는 코드를 한 번에 작성합니다 — 리뷰 없이, 감사 없이, 품질 게이트 없이. 동작은 합니다. 하지만 아무도 안 본 코드를 배포하시겠습니까?

**Harmony는 빠진 것을 채웁니다: 독립적 리뷰, 서버 검증 품질 게이트, 그리고 요구사항부터 프로덕션까지의 전체 개발 파이프라인.**

```
You: /harmony 인증과 결제가 있는 SaaS 대시보드 만들어줘

Harmony: 8개 핵심 질문 (a/b/c만 고르세요)
Harmony: 200줄 이상 PRD 작성 — 데이터 모델, API 계약 포함
Harmony: 당신의 기술 스택에 맞는 5인 에이전트 팀 구성
Harmony: 빌드 → 테스트 → 감사 → 수정 — 태스크별 자동 반복
Harmony: "12개 태스크 완료. 커버리지 87%. 보안 이슈 0건. 프로젝트 준비 완료."
```

> **승인 3번. 코드 0줄. 모든 코드가 독립적으로 감사되었습니다.**

## 왜 만들었나

모든 AI 코딩 도구에는 같은 맹점이 있습니다: **AI가 자기 숙제를 자기가 채점합니다.**

코드를 쓰고, 테스트를 돌리고, "완료"라고 합니다. 하지만 그 테스트가 의미 있는지 누가 검증했나요? 인증에 우회 경로가 없는지 누가 확인했나요? 에러 처리가 진짜 동작하는지 누가 봤나요?

Harmony는 이걸 세 가지 구조적 보장으로 해결합니다:

| 문제 | Harmony의 해법 |
|------|---------------|
| 작성자가 자기 코드를 리뷰 | **독립 감사자** — 새로운 에이전트, 공유 컨텍스트 없음 |
| 에이전트가 "테스트 통과"라고 보고 | **서버 교차검증** — 커버리지 도구를 독립적으로 실행 |
| 품질이 "권고사항" | **코드로 강제되는 게이트** — 임계값 미달 시 진행 불가 |

이건 프롬프트 엔지니어링이 아닙니다. [**하네스 엔지니어링**](https://martinfowler.com/articles/harness-engineering.html) — 저품질 출력을 구조적으로 어렵게 만드는 방법입니다. [설계 철학 전문 →](docs/DESIGN_PHILOSOPHY.md)

## 엔터를 누르면 일어나는 일

```
Phase 1 — 인터뷰                 8~12개 핵심 질문에 답하세요 (a/b/c)
Phase 2 — PRD 생성              스키마, API, 인수 조건이 포함된 200줄+ 스펙
Phase 3 — PRD 리뷰              승인, 수정, 또는 처음부터
Phase 4 — 셋업 (자동)           에이전트 팀 + 디자인 시스템 + 태스크 + 참조 문서
Phase 5 — 빌드 (자동)           수직 슬라이스 태스크마다:
                                  설계 문서 → 멀티 에이전트 동시 구현
                                  → 품질 게이트 (서버 검증)
                                  → 독립 프로덕션 감사
                                  → 모든 임계값 통과까지 수정 루프
Phase 6 — 검증                  새로운 QA 에이전트가 PRD의 모든 기능 구현 여부 확인
Phase 7 — 보안 강화             자동 스캐너 + AI 보안 분석
Phase 8 — 전달                  풀 빌드 + 통합 테스트 + 요약 리포트
```

**Phase 1~3에서 주로 개입합니다.** 이후에는 Harmony가 자율적으로 진행하며, 진짜 필요할 때만 멈춥니다.

## 품질 게이트 — 프롬프트로 제안이 아닌, 코드로 강제

메트릭은 에이전트가 자가 보고하는 게 아니라 서버가 측정합니다.

| 메트릭 | Prototype | MVP | Production |
|--------|-----------|-----|------------|
| 빌드 통과 | 필수 | 필수 | 필수 |
| 전체 테스트 통과 | 필수 | 필수 | 필수 |
| 린트 에러 0개 | 필수 | 필수 | 필수 |
| 테스트 커버리지 | >= 50% | >= 70% | >= 80% |
| 최대 파일 크기 | 600줄 | 400줄 | 350줄 |
| 최대 함수 크기 | 80줄 | 60줄 | 50줄 |
| 보안 크리티컬 | 0 | 0 | 0 |
| 접근성 크리티컬 | 0 | 0 | 0 |

단계는 인터뷰에서 설정됩니다. "Prototype"은 관대하고, "Production"은 엄격합니다. "건너뛰기" 버튼은 없습니다.

## 멀티 에이전트 팀

Harmony는 하나의 에이전트로 모든 걸 하지 않습니다. 당신의 기술 스택에 맞는 **프로젝트 전용 팀**을 구성합니다:

```
┌─────────────────────────────────────────────────────┐
│  Main Architect — 시스템 설계, 조율                    │
│  Code Architect — 설계 문서, 코드 구조                  │
│  DB Agent — 스키마, 마이그레이션, 쿼리 최적화             │
│  Review Agent — 독립적 코드 감사                       │
│  E2E Agent — 통합 테스트                              │
│  + 도메인 전문가 (프론트엔드, 백엔드 등)                  │
└─────────────────────────────────────────────────────┘
```

FastAPI 프로젝트와 Next.js 프로젝트는 다른 에이전트를 받습니다. 에이전트들은 **병렬 worktree**에서 동시 작업하고, 리뷰를 거쳐 머지하며, 해당 프로젝트에만 존재합니다.

각 에이전트는 도메인별 참조 문서를 받습니다 — 학습 데이터 기억이 아닌, 검증된 지식.

## 빌드 루프 (태스크별)

여기가 핵심입니다. 모든 태스크마다:

```
  ┌──────────────┐
  │   팀이 빌드   │  설계 문서 → 동시 구현 → 자기 리뷰
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │  품질 게이트   │  서버가 측정: 빌드, 테스트, 린트, 커버리지, 보안
  └──────┬───────┘
         ▼ 통과
  ┌──────────────┐
  │  독립 감사    │  새로운 에이전트. 공유 컨텍스트 없음. 코드 + PRD만 읽음.
  └──────┬───────┘
         ▼ 통과
  ┌──────────────┐
  │    완료      │
  └──────────────┘

  실패 → 수정 → 루프. 횟수 제한 없음.
  5라운드마다 사용자에게 에스컬레이션: 계속 / 수동 인수 / 중단
```

**책임감 압박**: 모든 에이전트는 다른 에이전트가 자기 작업을 냉정하게 평가할 것임을 압니다. 첫 품질이 달라집니다.

## 생성되는 구조

```
your-project/
├── docs/
│   ├── prd.md                     # 인터뷰 기반 전체 PRD
│   ├── refs/                      # 도메인 참조 문서 (할루시네이션 방지)
│   └── tasks/                     # 태스크별 설계 문서
├── .claude/agents/                # 프로젝트 전용 에이전트 팀
├── .harmony/state.json            # 파이프라인 상태 (재개 가능, git 추적)
├── CLAUDE.md                      # 프로젝트 규칙
├── src/                           # 실제 코드
└── tests/                         # 실제 커버리지 포함
```

## 세션 복원

레이트 리밋? 노트북 덮었다? 세션 크래시? 다시 실행하면 됩니다.

```bash
/harmony
# → "태스크 8/12에서 재개하시겠습니까? (a) 재개  (b) 처음부터  (c) 상태 확인"
```

**매 단계마다** 원자적 쓰기로 상태가 저장됩니다. 상태 파일은 git으로 추적되어 다른 컴퓨터에서도 이어할 수 있습니다.

## 적합한 프로젝트

| 잘 맞는 프로젝트 | 이유 |
|-----------------|------|
| 웹 앱 (Next.js, FastAPI, Django) | 예측 가능한 구조, 명확한 에이전트 역할 |
| API 서버 | 표준 패턴, 높은 테스트 가능성 |
| CLI 도구 | 단순한 구조, 빠른 파이프라인 |
| SaaS MVP | 인터뷰가 요구사항을 정확히 포착 |

| 덜 적합한 프로젝트 | 이유 |
|-------------------|------|
| 대규모 기존 코드베이스 (5만줄+) | 암묵적 의존성이 너무 많음 |
| ML / 데이터 파이프라인 | 실험적 워크플로우와 선형 태스크 불일치 |
| 하드웨어 / 임베디드 | 통합 테스트 자동화 곤란 |

## 명령어

| 명령어 | 설명 |
|--------|------|
| `/harmony [아이디어]` | **명령어 하나.** 아이디어 → 프로덕션 레디 프로젝트 |
| `/agent-harmony:harmony [아이디어]` | 정식 명칭 (항상 동작) |

내부 명령어 (파이프라인이 사용):

| 명령어 | 설명 |
|--------|------|
| `/project-init` | 새 프로젝트 스캐폴딩 |
| `/codebase-init` | 기존 코드베이스에서 초기화 |
| `/generate-agents` | 전용 에이전트 팀 생성 |
| `/build-refs` | 도메인 참조 문서 생성 |

## 알려진 동작

### 초기화 중 권한 모드 변경

프로젝트 초기화 시 Claude Code가 bypass 모드에서 "accept edits" 모드로 전환될 수 있습니다 — `.claude/`와 `.mcp.json` 파일에 대한 [플랫폼 레벨 보호](https://code.claude.com/docs/en/permission-modes.md)입니다.

**1회만 발생합니다.** 초기화 완료 후 `Shift+Tab`으로 bypass 모드로 복귀하세요.

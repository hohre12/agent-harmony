# Codebase Init Skill

Usage: `/codebase-init`

**Prerequisite**: A codebase to analyze must exist in the current directory. `docs/prd.md` is not required.

---

## Execution Order

### Step 1. Scan Codebase

Analyze the following in parallel:

**Map file structure:**
- Root directory structure (using `Glob`)
- Key config files: `package.json`, `pyproject.toml`, `pubspec.yaml`, `go.mod`, `Cargo.toml`, `Gemfile`, `pom.xml`, etc.
- Build/environment files: `Dockerfile`, `docker-compose.yml`, `.env.example`
- Existing documentation: `README.md`, files under `docs/`

**Detect tech stack:**
- Language: file extension distribution (`.py`, `.ts`, `.dart`, `.go`, etc.)
- Framework: extract from dependency files
- DB: ORM configs, migration file paths
- Cloud/external services: environment variable names, import patterns

**Identify architecture:**
- Directory layering (feature-based vs layer-based)
- API endpoint patterns
- Core domain models

### Step 1.5. Detect Git Structure and Base Branch

Check whether a `.git` folder exists in the root directory:

- `.git` exists at root → **monorepo mode** (root is the git project)
  - Run `git branch -r` or `git log --oneline` from root to determine the base branch
- `.git` does not exist at root → **multi-git mode** (sub-projects each have independent git repos)
  - Find the first sub-directory containing `.git` and determine the base branch from that directory

If the base branch cannot be determined, default to `main`.

> The detection result is used as a branching condition when generating CLAUDE.md in Step 7.

### Step 2. Generate architecture.md

Create `docs/` folder if it does not exist. Write `docs/architecture.md`:

```markdown
# {project name} Architecture

> Auto-generated from codebase analysis — {analysis date}

## Overview
{inferred project purpose — based on README or code structure}

## Tech Stack
| Category | Technology | Source File |
|----------|-----------|-------------|
| Language | ... | ... |
| Framework | ... | ... |
| DB | ... | ... |
| Cache | ... | ... |
| External Services | ... | ... |

## Directory Structure
{actual structure tree}

## Core Domains
{identified primary domains and modules}

## External Dependencies
{key packages extracted from package.json / pyproject.toml, etc.}

## Unverified Items
{aspects that were unclear during analysis — items the user should review and supplement}
```

### Step 3. Create Directory Structure

Create the following structure using a **merge approach**:

```
{project-root}/
├── .claude/
│   ├── agent-memory/
│   ├── agents/
│   └── skills/
└── docs/
    └── architecture.md  (just generated)
```

**Merge rules (when `.claude/` folder already exists):**

- Directory missing → create it
- Directory exists → keep it, enter and check contents
- File missing → create it
- File exists with same name → overwrite
- File exists with different name → leave untouched

> No user confirmation needed before overwriting. Merge silently.
> In the completion report, distinguish between "newly added" and "overwritten" items.

### Step 4. Generate .mcp.json

Select the necessary MCP servers based on codebase analysis and generate `.mcp.json`.

**Always included:**
```json
"context7": {
  "command": "npx",
  "args": ["-y", "@upstash/context7-mcp"]
}
```

**Conditionally included (based on codebase analysis):**

| Detected in codebase | MCP to add |
|---|---|
| UI/component libraries, design work artifacts | `stitch` |
| Supabase client imports | `supabase` |
| GitHub Actions, gh CLI usage | `github` |
| News scraping, web search logic | `exa` |
| Direct local filesystem manipulation | `filesystem` |
| Notion API usage | `notion` |

### Step 5. Generate CLAUDE.md

Create `CLAUDE.md` at the project root. Branch based on codebase analysis + **the Git structure detected in Step 1.5**:

**[monorepo mode]** (`.git` exists at root):

```markdown
# {project name}

{one-line project purpose inferred from codebase}

> This file was auto-generated by /codebase-init. Please manually correct any inaccuracies.

## Tech Stack
{detected tech stack}

## Project Structure
{actual directory structure summary}

## Git Structure
- Type: **monorepo**

## Development Rules
- Agent definitions: `.claude/agents/`
- Skill definitions: `.claude/skills/`
- Task management: managed by harmony pipeline (`.harmony/state.json`)
- Base branch: `{detected base branch — whichever of main/master/develop is actually in use}`
- Branch strategy: `feature/{tag}-{task-id}_{task-name}/{user}` (`{user}` = `whoami`)
- Branch chaining: if a previous task branch exists for the same tag, branch off of it (to include prior task code)
- Commits: use the `/commit` skill

## Reference Documents
- `docs/architecture.md`: Auto-generated architecture document
- `docs/refs/`: Per-agent domain reference documents (generated after running `/build-refs`)

## Language
All responses must match the language used in the user's initial prompt.
```

**[multi-git mode]** (`.git` does not exist at root):

```markdown
# {project name}

{one-line project purpose inferred from codebase}

> This file was auto-generated by /codebase-init. Please manually correct any inaccuracies.

## Tech Stack
{detected tech stack}

## Project Structure
{actual directory structure summary}

## Git Structure
- Type: **multi-git** (root is not a git project)
- Sub-project list:
  - `{sub-project-dir}/` — {responsible domain}  ← list each sub-project identified from codebase here

## Development Rules
- Agent definitions: `.claude/agents/`
- Skill definitions: `.claude/skills/`
- Task management: managed by harmony pipeline (`.harmony/state.json`)
- Run Claude: from the root folder
- Branch creation/commits/pushes: run inside each sub-project
  - Base branch: `{detected base branch — whichever of main/master/develop is actually in use}`
  - Branch strategy: `feature/{tag}-{task-id}_{task-name}/{user}` (`{user}` = `whoami`)
  - Branch chaining: if a previous task branch exists for the same tag, branch off of it (to include prior task code)
  - Commits: use the `/commit` skill
- **Caution**: The root folder is not a git project. Git commands must be run inside sub-project directories.

## Reference Documents
- `docs/architecture.md`: Auto-generated architecture document
- `docs/refs/`: Per-agent domain reference documents (generated after running `/build-refs`)

## Language
All responses must match the language used in the user's initial prompt.
```

### Step 6. Copy expert-agent.md

Copy the plugin's expert-agent into the project. Search in this order:

1. Plugin directory: find the `agent-harmony` plugin and copy `agents/expert-agent.md`
2. Fallback: `~/.claude/agents/expert-agent.md`

```bash
# The expert-agent.md is bundled with the agent-harmony plugin.
# Use Glob to find it: ~/.claude/plugins/agent-harmony/agents/expert-agent.md
# or a similar plugin installation path.
# Copy it to the project:
cp {found_path}/agents/expert-agent.md .claude/agents/expert-agent.md
```

If expert-agent.md cannot be found in any location, abort immediately and notify the user.

### Step 7. Generate .claude/settings.local.json

Create the settings file to enable agent team features and permissions:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "permissions": {
    "allow": [
      "Bash(*)",
      "Edit(*)",
      "Read(*)",
      "WebFetch(*)",
      "WebSearch"
    ],
    "defaultMode": "bypassPermissions"
  },
  "teammateMode": "auto"
}
```

> Without this file, the `team-executor` skill's team features (`TeamCreate`, `SendMessage`, etc.) will not work.

## Rules

- Exclude `.git/`, `node_modules/`, `.venv/`, `__pycache__/` when scanning the codebase
- Even if `.claude/` already exists, proceed with **merge approach** without asking for overwrite confirmation
  - Same filename → overwrite
  - Different filename → keep existing file
  - Missing folder/file → create new
- If `.mcp.json` already exists → overwrite (single file)
- When generating `docs/architecture.md`, clearly distinguish between inferred and confirmed information
- MCP server selection must be justified with evidence from the codebase
- After all files are created, output the structure in tree format
- In the completion report, distinguish between "newly added", "overwritten", and "preserved (existing file kept as-is)" items
